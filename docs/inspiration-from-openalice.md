## 1. Project Overview

**OpenAlice** calls itself "your one-person Wall Street" — an AI trading agent covering
equities, crypto, commodities, forex, and macro. It's a TypeScript/Node.js monorepo
(pnpm workspaces + Turborepo) that runs on your own machine.

**Our project** is a focused Rust option trading CLI for Interactive Brokers:
- Single binary (no Node.js runtime needed)
- TUI + headless CLI (not AI agent)
- Options-only focused (not multi-asset)

## 3. What We DON'T Build

OpenAlice wrote a **full TWS wire protocol implementation from scratch** in TypeScript.
That's exactly what we avoid by using the `ibapi` Rust crate.

| OpenAlice Did | We Skip Because | `ibapi` Provides |
|---|---|---|
| TCP socket with length-prefixed message framing | TWS protocol handled | Built-in `ibapi::Client` |
| 150+ protobuf message type definitions | Protocol generated by `ibapi` | v3.0 protobuf, auto-synced with IBKR updates |
| Request ID multiplexer with pending Promise map | Built into `ibapi` | `Subscription<T>` streaming model |
| Raw message encoder/decoder per message type | Client handles serialization | Fluent API (`client.order(&c).buy(1).market()`) |
| Server version compatibility checks (50+ checks in placeOrder) | Handled internally | Version negotiation + compatibility guards |
| Order wire format encoding (~450 lines of field serialization) | Handled internally | `client.placeOrder(order_id, contract, order)` |

**Bottom line:** OpenAlice's `packages/ibkr` is proof that the TWS protocol is implementable,
but it's ~10,000 lines of message-level code that we don't need to write.

## 4. Patterns We Adopt

These are architectural patterns from OpenAlice's **service/uta layer** (not the protocol layer)
that directly influence our Rust design.

### 4.1 Contract Builder with Validation (contract-builder.ts)

**OpenAlice pattern:** A single `buildContract()` function that takes clean input,
constructs a `Contract`, and validates required fields per security type via `assertContract()`.

```ts
// OpenAlice: validated contract construction
buildContract({
  symbol: "SPX", secType: "OPT", exchange: "SMART", currency: "USD",
  strike: 4000, right: "C", lastTradeDateOrContractMonth: "20250117",
  multiplier: "100"
})
// → Throws if OPT is missing strike, right, or lastTradeDateOrContractMonth
```

**Our Rust equivalent (planned):**
```rust
// Builder with compile-time and runtime validation per sec type
Contract::builder()
    .symbol("SPX")
    .sec_type(SecType::Opt)
    .exchange("SMART")
    .currency("USD")
    .strike(4000.0)
    .right(Right::Call)
    .expiry("20250117")
    .build()?;  // Runtime: validates OPT has strike+right+expiry
```

**Why:** Prevents silent contract errors that surface as "contract not found" at order time.

### 4.2 IBKR Error Classification (ibkr-contracts.ts)

**OpenAlice pattern:** Maps IBKR's numeric error codes to typed, actionable error categories.

```ts
// OpenAlice: error classification
classifyCode(502)  // → 'NETWORK' — "Couldn't connect to TWS"
classifyCode(326)  // → 'AUTH' — "Client ID already in use"
classifyCode(200)  // → 'EXCHANGE' — "No security definition found"
classifyCode(399)  // → 'MARKET_CLOSED' — if message contains "outside trading hours"
```

**Our Rust equivalent (planned):**
```rust
// In error.rs
#[derive(Debug, thiserror::Error)]
pub enum AppError {
    #[error("Network error: {0}")]
    Network(String),
    #[error("Authentication error: {0}")]
    Auth(String),
    #[error("Exchange rejected: {0}")]
    Exchange(String),
    #[error("Market closed: {0}")]
    MarketClosed(String),
}

impl AppError {
    pub fn from_ibkr_code(code: i32, msg: &str) -> Self {
        match code {
            502 | 504 | 1100..=1102 => Self::Network(msg.into()),
            326 => Self::Auth(msg.into()),
            200..=210 | 103..=110 => Self::Exchange(msg.into()),
            _ if msg.contains("outside") && code == 399 => Self::MarketClosed(msg.into()),
            _ => Self::Unknown(format!("IBKR error {}: {}", code, msg)),
        }
    }
}
```

**Why:** A TUI that shows "Exchange rejected: Order rejected - price does not conform to min tick"
is vastly more useful than "Error: code 110".

### 4.3 Clean Broker Interface (IBroker trait)

**OpenAlice pattern:** A trait/interface that abstracts any broker behind the same operations.

```ts
interface IBroker {
  init(): Promise<void>
  close(): Promise<void>
  getAccount(): Promise<AccountInfo>
  getPositions(): Promise<Position[]>
  getQuote(contract): Promise<Quote>
  placeOrder(contract, order): Promise<PlaceOrderResult>
  modifyOrder(orderId, changes): Promise<PlaceOrderResult>
  cancelOrder(orderId): Promise<PlaceOrderResult>
  getCapabilities(): AccountCapabilities
}
```

**Our Rust equivalent (planned):**
```rust
#[async_trait]
pub trait Broker: Send + Sync {
    async fn connect(&mut self, config: &ConnectionConfig) -> Result<()>;
    async fn disconnect(&mut self) -> Result<()>;
    async fn get_account(&self) -> Result<AccountSummary>;
    async fn get_positions(&self) -> Result<Vec<Position>>;
    async fn get_quote(&self, contract: &OptionContract) -> Result<Quote>;
    async fn place_order(&self, contract: &OptionContract, order: &Order) -> Result<OrderId>;
    // ...
}
```

**Why:** Even with only IBKR, the trait forces clean API boundaries. If we ever add
multi-leg spread building, the interface keeps the TUI layer broker-agnostic.

### 4.4 Multiplier-Aware Position Math (position-math.ts)

**OpenAlice pattern:** Force multiplier to always be populated for OPT/FOP positions,
and catch `multiplier=1` as a bug (would underreport values by 100x).

```ts
// OpenAlice caught this bug:
if ((secType === 'OPT' || secType === 'FOP') && multiplier === '1') {
  throw new Error("Upstream broker decode missing multiplier — values would be 100x wrong")
}
```

**Why:** This is a real bug pattern. TWS doesn't always send the multiplier in all
messages. Always fetch full `ContractDetails` to get the real multiplier.

---

### 4.5 Request Bridge Pattern (request-bridge.ts)

**OpenAlice pattern:** Single TCP socket with multiple pending requests identified by `reqId`.

```ts
// reqId-based routing:
pending.set(reqId, { resolve, reject, timer })
client.reqContractDetails(reqId, contract)  // sends on wire
// ... response comes back on same socket ...
// wrapper.route(reqId) → finds pending promise → resolves it
```

**Why for us:** While `ibapi` handles this internally, understanding this pattern
is essential for debugging why our requests might timeout or get misrouted.

## 5. What OpenAlice Leaves for Us

OpenAlice is an **AI agent runtime**, not a CLI trading tool. These gaps are our opportunity.

| Feature | OpenAlice | Our CLI |
|---|---|---|
| **TUI option chain** | ❌ No terminal UI at all | ✅ ratatui-based interactive table with scrolling, color-coding, ATM highlight |
| **Real-time streaming** | ❌ `getQuote()` is snapshot-only | ✅ tick-by-tick bid/ask subscription → live-updating chain table |
| **Keyboard order entry** | ❌ AI agent places orders via MCP tools | ✅ `tui-textarea` form with Tab navigation, Enter to submit |
| **Headless CLI** | ❌ No CLI subcommands | ✅ `ibkr chain/watch/trade/positions/status` |
| **Single binary** | ❌ Requires Node.js, pnpm, 1.7GB install | ✅ Static binary, ~10MB, zero runtime deps |
| **Option chain table** | ❌ Only has `getContractDetails()` per contract | ✅ `ibapi.option_chain()` → full expiry/strike grid |
| **Connection management** | ❌ Manual restart on disconnect | ✅ Auto-reconnect with exponential backoff |
| **Configuration** | ❌ No config file | ✅ `config.toml` with host/port/client-id |
